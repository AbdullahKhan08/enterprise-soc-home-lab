# IR-001: Suspicious PowerShell Execution Using Execution Policy Bypass

## Incident Summary

A Wazuh level 10 alert detected PowerShell execution using the `-ExecutionPolicy Bypass` parameter on the monitored Windows endpoint.

The activity was generated as a safe, authorized lab simulation to validate the custom suspicious PowerShell parameter detection rule.

## Detection Logic

The alert was generated when Wazuh detected a PowerShell process creation event containing suspicious execution parameters.

The custom rule inspected the Sysmon command line for indicators including:

- `-ExecutionPolicy Bypass`
- `-EncodedCommand`
- `-WindowStyle Hidden`

## Triggering Command

The following safe command was intentionally executed in the authorized home lab:

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -Command "Get-Date"
```

## Alert Evidence

The Wazuh alert contained the following evidence:

- Windows endpoint hostname
- User context
- PowerShell executable path
- Full process command line
- `-ExecutionPolicy Bypass` parameter
- Parent process details
- Process GUID and Process ID
- SHA-256 hash
- Wazuh Rule ID `100101`
- Alert severity level `10`
- MITRE ATT&CK mapping to `T1059.001`

## Investigation Process

1. Reviewed the Wazuh alert generated by Rule `100101`.
2. Confirmed the suspicious parameter was visible in the Sysmon command-line telemetry.
3. Reviewed the endpoint, user context, parent process, and command-line details.
4. Confirmed the command was executed manually as part of an authorized detection-validation test.
5. Verified there was no malicious payload, persistence mechanism, credential access attempt, external download, or suspicious network activity associated with the command.

## Root Cause

The alert was triggered by an intentional lab test using `-ExecutionPolicy Bypass` to validate the custom Wazuh detection rule.

## Impact Assessment

No security impact was identified.

The execution was limited to a benign `Get-Date` command inside the isolated SOC lab environment. No unauthorized changes, malware execution, data access, persistence, or network-based activity occurred.

## Response Actions

- Confirmed the event was authorized.
- Documented the event as a successful detection validation.
- Retained the detection rule because execution-policy bypass remains a high-value investigation signal.
- Closed the incident as authorized test activity.

## Analyst Notes

`-ExecutionPolicy Bypass` is not automatically malicious. However, it is frequently used by attackers to bypass PowerShell execution restrictions.

In a real enterprise environment, an analyst should investigate:

- Whether the user is authorized to run PowerShell with bypass parameters
- Parent process and process chain
- Script block contents
- Downloaded files or payloads
- Network connections made by PowerShell
- Persistence mechanisms created after execution
- Similar activity across other endpoints
- Whether the command is associated with approved automation or management tools

## Lessons Learned

- Sysmon Event ID 1 provides valuable process creation and command-line telemetry.
- PowerShell Script Block Logging provides deeper visibility into PowerShell activity.
- Detection chaining improves detection quality: Rule `100101` builds on the PowerShell process-creation rule `100100`.
- A high-severity alert still requires analyst validation before being classified as malicious.
- Documenting benign tests is useful for proving both technical detection capability and SOC investigation workflow.
